Inode numbers of all files
Command: ls -li original.txt hardlink.txt symlink.txt
Explanation: I ran ls -li to view the index allocation numbers (inodes) for all three references. The output shows that original.txt and hardlink.txt share the exact same inode number (524), while symlink.txt points to its own unique inode (5419).

Comparison between hard link and symbolic link
Commands:
ln original.txt hardlink.txt
ln -s original.txt symlink.txt
Explanation: I created a hard link and a symbolic link to compare how they track the source file. The hard link creates a direct pointer to the existing data on the disk, while the symbolic link works like a shortcut, simply pointing to the text filename of the original file.

Effect of deleting the original file
Commands:
rm original.txt
cat hardlink.txt
cat symlink.txt
Explanation: After deleting original.txt, running cat hardlink.txt still worked and printed out the text because its link to inode 524 remained intact. However, cat symlink.txt broke and returned an error because its target path string was gone, creating a broken link.

Metadata collected using Linux utilities
Command: ls -li original.txt hardlink.txt symlink.txt
Explanation: Looking closely at the ls metadata, the link count for inode 524 was initially 2, showing that two filenames pointed to that same data block. The file size of the symlink is also smaller (12 bytes) because it only contains the text string of the target path.

Conclusions drawn from the experiment
Explanation: This experiment proves that hard links share the exact same data source and continue working independently even if the original filename is deleted. On the other hand, symbolic links are completely dependent on the original filename and become useless if the source file is moved or removed.
